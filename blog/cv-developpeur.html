CV développeur 2026 : guide complet + modèle annoté senior Full-Stack

📅 23 mai 2026 ⏱ 12 min de lecture 👁 Emploia Editorial

Un tech lead passe 30 secondes sur votre CV avant de décider si vous méritez un entretien. Dans ces 30 secondes, il cherche trois choses : votre stack technique, vos projets avec impact mesurable, et votre niveau de séniorité. Pas votre photo, pas vos hobbies, pas vos barres de compétences en étoiles.

Ce guide vous donne le modèle exact, les formulations qui fonctionnent, et les erreurs qui font immédiatement rejeter un CV développeur — même excellent sur le fond.

Ce que le recruteur technique cherche en 30 secondes

Contrairement aux recruteurs RH, le tech lead qui lit votre CV est lui-même développeur. Il repère instantanément les red flags et les signaux positifs :

✅ Signaux positifs : stack précis + version (React 18, Node 20, TypeScript 5), chiffres d'impact (−40% temps de chargement, 99.9% uptime), contributions open source, projets avec utilisateurs réels, GitHub actif.

🚩 Red flags immédiats : "maîtrise de nombreux langages", barres de compétences visuelles, "passionné de nouvelles technologies", stack générique sans version, expériences sans aucun chiffre.

Tableau par séniorité : ce qui doit apparaître sur votre CV

NiveauStack à mentionnerType de projetsBullet point type
Junior (0–2 ans) Langages maîtrisés + frameworks principaux Projets personnels, stage, contributions open source "Développé une API REST en Node.js/Express avec authentification JWT"
Mid (2–5 ans) Stack complète + outils CI/CD + cloud Features en production, refactoring, optimisations "Réduit le temps de réponse API de 340ms à 95ms via cache Redis"
Senior (5–10 ans) Architecture + choix technologiques + mentoring Architecture système, migration, leadership technique "Migré monolithe vers microservices (8 services), −60% temps de déploiement"
Tech Lead / Principal Stack + décisions d'architecture + équipe Design système, roadmap technique, recrutement "Défini architecture event-driven (Kafka), guidé équipe de 6 devs sur 18 mois"

Modèle de CV annoté — Senior Full-Stack React/Node

📋 Accroche (3 lignes max)

"Développeur Full-Stack senior (6 ans) spécialisé React 18 / Node.js / TypeScript. Expérience sur des applications SaaS à fort trafic (500k+ utilisateurs). Orienté performance et code maintenable — contributeur open source (2,3k⭐ GitHub)."

🛠 Stack technique — format recommandé

  • Frontend : React 18, Next.js 14, TypeScript 5, Tailwind CSS, Zustand
  • Backend : Node.js 20, Express, NestJS, REST, GraphQL, WebSocket
  • Base de données : PostgreSQL, MongoDB, Redis, Prisma ORM
  • DevOps : Docker, Kubernetes, GitHub Actions, AWS (EC2, S3, Lambda), Vercel
  • Tests : Jest, Vitest, Playwright, Testing Library

💼 Expériences — structure recommandée

Développeur Full-Stack Senior — Startup SaaS B2B (2022–2026)

  • Architecturé et développé le module de facturation (Stripe, webhooks, 15k abonnés) — réduit les erreurs de paiement de 23%
  • Migré l'app React class components → hooks + Context API → −40% de re-renders, bundle size −28%
  • Mis en place monitoring (Datadog, alertes Slack) — MTTR passé de 45min à 8min
  • Onboardé et mentoré 2 développeurs juniors (revues de code hebdomadaires, pair programming)

Avant / après : les formulations qui font la différence

❌ Avant — vague

"Développement de fonctionnalités pour une application web en React"

✅ Après — avec impact

"Développé le module de recherche full-text (Elasticsearch) — requêtes passées de 1.2s à 180ms, +34% d'engagement"

❌ Avant — générique

"Participation à la mise en place de tests automatisés"

✅ Après — précis

"Élevé la couverture de tests de 12% à 78% (Jest + Playwright) — zéro régression en production sur 8 mois"

❌ Avant — responsabilités

"Responsable de l'optimisation des performances de l'application"

✅ Après — résultats

"Optimisé LCP de 4.2s à 1.1s (lazy loading, code splitting, CDN) — score Lighthouse 42→94"

Les barres de compétences : pourquoi c'est une erreur

Ne jamais utiliser de barres visuelles (étoiles, pourcentages, jauges) pour les compétences techniques.

Problème 1 : les ATS ne les lisent pas. Problème 2 : un tech lead qui lit "Python ⭐⭐⭐⭐☆ (80%)" ne sait toujours pas si vous êtes capable de faire du data engineering ou juste du scripting basique. Problème 3 : vous vous comparez à vous-même, pas à un référentiel objectif.

Remplacez par : une liste structurée avec des exemples concrets. "Python (FastAPI, pandas, 3 projets en production)" est infiniment plus utile.

Projets personnels et open source

Si vous avez moins de 5 ans d'expérience, les projets personnels comptent autant que vos expériences pro. Un junior avec un projet GitHub à 800 étoiles sera systématiquement préféré à un junior sans.

Format recommandé pour un projet perso :

NomDuProjet — github.com/vous/projet | 1.2k ⭐ | 3.2k utilisateurs actifs

SaaS de gestion de newsletters construit avec Next.js 14, Prisma, Resend. Architecture serverless déployée sur Vercel Edge. MRR : 120€.

Règles ATS spécifiques au CV développeur

  • Évitez les tableaux Word pour votre stack : les ATS les lisent mal. Utilisez une liste simple.
  • Écrivez les noms de technos exactement comme sur les offres : "React.js" et "React" ne sont pas forcément le même token pour un ATS.
  • N'utilisez pas de colonnes : un CV en deux colonnes avec votre stack à gauche perd 40% de son contenu dans les ATS basiques.
  • Incluez les acronymes ET les formes complètes : "CI/CD (GitHub Actions, GitLab CI)" pour matcher les deux formes de recherche.
  • PDF exporté depuis Word ou Google Docs : évitez les PDF "image" — le texte doit être sélectionnable.

Questions fréquentes

Un CV développeur doit-il faire 1 ou 2 pages ?
1 page jusqu'à 5 ans d'expérience, 2 pages maximum pour un senior/lead. Si vous avez 8 ans d'expérience et ne pouvez pas tenir en 2 pages, vous listez des responsabilités au lieu de résultats — condensez.
Faut-il mettre GitHub sur son CV ?
Oui, systématiquement — mais seulement si votre profil est actif et présente du code de qualité. Un GitHub vide ou avec des projets abandonnés depuis 4 ans fait plus de mal que bien. Fixez vos repos avant de postuler.
Comment mettre en avant des technologies que j'apprends ?
Ajoutez une section "En apprentissage" ou "Side projects" avec un projet concret. "Kubernetes (en cours — déploiement d'une app Next.js sur cluster GKE)" est honnête et montre votre curiosité. Ne listez jamais une techno que vous ne sauriez pas défendre en entretien technique.
